# LazyFirst 项目整体优化方案

> 文档性质：纯分析与建议，**不含任何代码改动 / 数据库改动 / 文件删除**。
> 生成日期：2026-07-14
> 范围：后端 `server/` + MongoDB 数据库 + 前端 `frontend/` + Chrome 插件 `tiktok-extension/`
> 说明：本方案所有"归档 / 删除 / 合并 / 迁移"均为建议，实际执行需主人另行授权；涉及生产库的操作必须经主人同意。

---

## 1. 项目现状总览

### 1.1 技术栈

- 后端：Node.js + Express + Mongoose 8 + JWT（`jsonwebtoken`）+ `express-validator` + `helmet` + `express-rate-limit`
- 数据库：MongoDB（多租户按 `companyId` 隔离；`mongoose-memory-server` 仅作为 devDependency，生产用真实 MongoDB）
- 前端：Vue 3 + Element Plus + Vue Router + Pinia（`stores/`）
- 插件：Chrome 扩展（Manifest V3），独立子项目
- 部署：Docker（`docker-compose.yml`、`Dockerfile`、`Dockerfile.frontend`）+ Nginx（`nginx.conf`）

### 1.2 模块地图与代码量（估算）

| 域 | 目录 | 规模 |
|---|---|---|
| 后端模型 | `server/models/` | 34 个 `.js`（含 `index.js`） |
| 后端路由 | `server/routes/` | 34 个 `.js` |
| 后端服务层 | `server/services/` | 1 个（`volcano-api.js`） |
| 后端中间件 | `server/middleware/` | 4 个 |
| 后端脚本 | `server/scripts/`（12）+ `server/archived-migrations/`（14）+ 根目录散落（7） | 33 个脚本 |
| 前端视图 | `frontend/src/views/` | 约 46 个 `.vue` |
| 前端组件 | `frontend/src/components/` | 若干 |
| 前端 i18n | `frontend/src/i18n/` | zh.js / en.js |
| 插件 | `tiktok-extension/` | 33 个文件（11 js / 3 md / 4 png / 1 svg / 其他） |

### 1.3 核心结论

- 功能完整、业务闭环清晰，但**代码组织技术债较重**：模型未统一导出、超大路由文件、service 层几乎空白、路由内 try/catch 泛滥、大量一次性/归档脚本散落。
- **数据库文档已严重过时**：`database-schema.md` 只记录 24 个模型，实际 34 个；多张表存在"兼容旧字段"冗余与个别悬空引用。
- **历史遗留产物堆积**：同步记录、归档迁移、备份 json、`dist` 构建产物、log、图片等未归置。
- **插件与后端契约存在断裂点**：批量视频保存接口 404、后端地址硬编码四处不一致。

### 3.2 逐表字段级冗余与问题

#### 3.2.1 `Order` —— 兼容字段遗留（高风险悬空引用）

文件：`server/models/Order.js`

| 字段 | 问题 | 证据 |
|---|---|---|
| `storeId` | `ref: 'Store'` —— **`Store` 模型不存在**（悬空引用） | L182-185 |
| `activityId` | `ref: 'Activity'`，但导入路由未写入 | L186-189 |
| `orderDate` / `products[]` / `totalAmount` / `commissionRate` / `currency` / `status` | 注释"原有字段保持兼容"，**导入流程不写入**，纯遗留 | L181-217 |

**建议**：标记 `storeId`/`activityId`/`orderDate`/`products`/`totalAmount`/`commissionRate` 为废弃；`storeId` 因引用不存在模型应优先清理。需先确认 `orders.js` 路由是否仍有写入/读取这些字段（小垃圾已确认导入路由未写入），再评估是否整体废弃 `Order` 或收敛为仅保留报表用途的 `ReportOrder`。

#### 3.2.2 `ReportOrder` vs `Order` —— 字段大面积重复（需明确职责边界）

两者共享：`companyId` + 全部 Excel 导入字段（`orderNo`/`subOrderNo`/`influencerUsername`/`productId`/`productName`/`sku`/`productPrice`/`orderQuantity`/`shopName`/`shopCode`/`orderStatus`/`contentType`/`contentId`）+ 6 种佣金率 + 7 种预计佣金 + 7 种实际佣金 + 退货退款（`returnedProductCount`/`refundedProductCount`）+ 时间字段 + 打款字段（`paymentNo`/`paymentMethod`/`paymentAccount`）+ `iva`/`isr`/`platform`/`attributionType`。

差异：
- `Order` 独有：`influencerId`、`storeId`、`activityId`、`orderDate`、`products[]`、`totalAmount`、`commissionRate`、`currency`、`status`
- `ReportOrder` 独有：`userId`、`summaryDate`、`merchandiser`、`bdName`、`orderCount`、`gmv`、`groupInfo`、`country`、`isBlacklistedInfluencer`、`settlementStatus`、`settlementBillNo`

**建议**：当前为"订单原始导入表 + 报表去重表"的范式，`ReportOrder` 承担报表/结算/账单联动。优化方向二选一（需主人定夺）：
1. 保留双表，但在文档中明确职责边界，避免后续开发继续往 `Order` 写重复字段；
2. 评估 `Order` 是否还有独立消费方，若无则逐步收敛为单表（`ReportOrder` 作为唯一订单表）。

> 注意：唯一索引不同——`ReportOrder` 为 `{orderNo, subOrderNo, companyId}` 唯一；`Order` 无唯一约束。合并前必须先解决主键/去重语义。

#### 3.2.3 `SampleManagement` —— 三组"兼容旧字段"并存

文件：`server/models/SampleManagement.js`

| 新字段（规范） | 兼容旧字段 | 建议 |
|---|---|---|
| `sampleStatus`（`pending/shipping/sent/refused`） | `isSampleSent`（Boolean，"兼容旧数据"） | 以 `sampleStatus` 为准；迁移旧 `isSampleSent=true` → `sent` 后**废弃 `isSampleSent`**（同时移除其索引） |
| `influencerId`（ObjectId） | `influencerAccount`（String） | 新数据走 `influencerId`；旧数据保留读取，新功能不再写 `influencerAccount` |
| `salesmanId`（ObjectId） | `salesman`（String） | 同上，统一为 `salesmanId` |

**建议**：存量数据迁移脚本（仅建议在本地/测试库验证后、经主人授权在生产执行）将 `isSampleSent` 映射到 `sampleStatus`，随后在模型中删除 `isSampleSent` 及相关索引。

#### 3.2.4 `Influencer` —— 反范式快照字段（设计取舍）

`Influencer` 上保留了 `latestFollowers`/`latestGmv`/`latestMaintenanceTime`/`latestMaintainerId`/`latestMaintainerName`/`latestRemark`，与 `InfluencerMaintenance` 明细表存在冗余。

**建议**：此为**有意的反范式缓存**（避免每次读取达人列表都聚合最新维护记录），性能上合理。优化方向：
- 保留现状，但在 `database-schema.md` 中明确标注"快照字段，由维护记录写入时同步更新"；
- 或改为读取时聚合（`$last`/`$sort`），前提是有索引支撑且列表量可控。当前**建议保留**，仅补文档。

#### 3.2.5 `Commission.sampleRequestId` —— 悬空引用

`server/models/Commission.js` 中 `sampleRequestId` 的 `ref: 'SampleRequest'`，但项目无 `SampleRequest` 模型（样品相关模型为 `SampleManagement`）。

**建议**：改为 `ref: 'SampleManagement'`（若语义为样品申请），或置为可选/null。属低风险的字段修正。

#### 3.2.6 `Shop.products` vs `Product.shopId` —— 双向冗余

`Shop` 新增 `products: [String]`（文档未记录），与 `Product.shopId`（ref Shop）构成双向关联。

**建议**：以 `Product.shopId` 为唯一归属来源，`Shop.products` 改为查询时反向聚合（或在文档标注其为缓存、需与 `Product` 保持一致）。避免两处同时维护导致不一致。

#### 3.2.7 `Activity.sampleMethod` 枚举不一致

`Activity` 的 `sampleMethod` 枚举为 `['线上','线下','']`，而 `Product.activityConfigs[].sampleMethod` 使用 `online/offline`，语义相同但取值不同，易在联动逻辑出错。

**建议**：统一枚举取值（推荐 `online/offline`），或建立映射层，避免跨表不一致。

#### 3.2.8 `TempIdMapping` —— 导入临时表

仅 `init-import.js` 导入 Excel 时建立"原始 ID → 新 ObjectId"映射，导入完成后无持续用途。

**建议**：导入成功后按 `companyId` 清理；或加 TTL 索引自动过期（如 7 天）。可避免该表无限增长。

### 3.3 索引优化建议

| 模型 | 现状 | 建议 |
|---|---|---|
| `Order` | 无唯一索引；`storeId`/`activityId` 悬空 | 若保留，补 `{companyId, orderNo, subOrderNo}` 唯一；清理悬空索引 |
| `ReportOrder` | 已有 3 个索引 + 唯一索引 | 合理，保留 |
| `SampleManagement` | 唯一索引已改 `{companyId,date,influencerId,productId}` + 多个辅助索引 | 移除 `isSampleSent` 索引（随字段废弃） |
| `Influencer` | 5 个复合索引 + `maintenanceStatus` 虚拟字段 | 虚拟字段无法建索引；若按状态过滤频繁，考虑冗余物理字段 |
| 全局 | 所有表均按 `companyId` 建索引 | 一致，保留 |

### 3.4 表级结论速览

| 表 | 结论 |
|---|---|
| `Order` | 收敛/评估是否合并到 `ReportOrder`；清理悬空 `storeId`/`activityId` |
| `ReportOrder` | 保留（报表/结算主表） |
| `SampleManagement` | 保留；废弃 3 组兼容字段 |
| `Influencer` | 保留；补文档说明快照字段 |
| `Commission` | 修正 `sampleRequestId` 悬空引用 |
| `Shop` / `Product` | 解决 `products`/`shopId` 双向冗余 |
| `Activity` | 统一 `sampleMethod` 枚举 |
| `TempIdMapping` | 导入后清理 / 加 TTL |
| 其余 26 个表 | 文档补齐即可，无显著冗余 |

## 4. 后端逻辑与代码优化建议

### 4.1 模型导入风格统一

- `server/models/index.js` 仅导出 17/34 个模型，其余（含高频使用的 `Bill`、`ReportOrder`、`BaseData`、`BdDaily`、`Activity`、`AiModel` 等）只能在路由里直接 `require`。
- `report-orders.js` 在函数体内重复 `require('../models/Bill')` 达 13 次、`require('../models/ReportOrder')` 达 10 次（本可在文件顶部一次性引入）。
- **建议**：
  1. 将全部 34 个模型统一在 `models/index.js` 导出，路由改为 `const { Bill, ReportOrder } = require('../models')`；
  2. 全量扫描并消除函数体内重复 `require`（重点 `report-orders.js`、`product-stats.js`、`videos.js`、`bd-daily.js`）。

### 4.2 超大路由文件拆分

| 文件 | 约行数 | 建议拆分 |
|---|---|---|
| `report-orders.js` | ~1700 | 拆为 `reportOrders.query.js`（列表/搜索/分页/权限）、`reportOrders.import.js`（Excel 解析）、`reportOrders.bdMatch.js`（BD 匹配算法）、`bills.js`（账单生成/结算/备注/支付记录，约 11 端点独立成路由） |
| `samples.js` | ~1200 | 拆为 `samples.crud.js` + `samples.import.js` + `samples.public.js` |
| `product-stats.js` | ~1000 | 抽离 `services/productStats.js` 承载聚合计算 |
| `init-import.js` | ~850 | 抽离 `services/import/*.js` |
| `activities.js` / `bd-daily.js` / `influencer-managements.js` | 580–700 | 提取公共聚合/权限逻辑到 service |

### 4.3 service 层补齐

- 现状：`server/services/` 仅 `volcano-api.js`（第三方封装），业务计算全堆在路由里。
- **建议**：优先将以下重逻辑下沉到 service 层（同时消除"路由过胖"与"重复 require"）：
  - `report-orders.js` 的 BD 匹配算法（视频匹配 / 样品匹配）、Excel 导入字段映射；
  - `product-stats.js` 的 overview 聚合管道；
  - 通用：Excel 解析、`parseExcelDate`、账单生成。

### 4.4 统一错误处理，削减 try/catch

- `report-orders.js` 17 处、`samples.js` 16 处、`init-import.js` 14 处 try/catch，几乎每个 handler 独立包裹，错误格式不统一。
- **建议**：已有 `middleware/errorHandler.js`，改为 Express 异步错误传播（`async handler` + `next(err)`），由统一 `errorHandler` 兜底；路由内移除冗余 try/catch，统一错误响应结构。

### 4.5 中间件与权限

- `middleware/` 4 个文件职责清晰、无重复，保持。
- `auth.js` 的 `filterByDataScope` 已在多处使用，建议在 service 层复用而非每路由重写数据权限过滤。

### 5.4 i18n 一致性

## 6. tiktok-extension 优化建议

### 6.1 结构现状

- Manifest V3，`background.service_worker` + `content_scripts`（`content.js`/`injected.js` + 2 个 Overlay 组件）+ `popup` + `options`。
- 核心功能：TikTok 页面识别、达人/视频指标抓取（读 `SIGI_STATE`/`__RENDER_DATA__` 等内部全局）、单达人/批量视频保存、历史指标查询、私信模板。

### 6.2 与后端契约问题（阻塞/高风险）

| 问题 | 证据 | 影响 |
|---|---|---|
| **`POST /batch-videos` 后端不存在** | `background.js:240` 调用；`tiktok-extension-data.js` 无此路由 | 批量保存视频必然 404，批量导入功能实际失效 |
| **后端 base URL 硬编码 4 处不一致** | `constants.js:6` `localhost:3001/api`、`popup.js:9` `localhost:5174`、`background.js:98` `https://lazyfirst.com`、`options.js:10` `https://tap.lazyfirst.com` | 本地/生产地址各自为政 |
| **`/api` 后缀靠正则 hack** | `background.js:99` 与 `popup.js:23` 均 `replace(/\/api\/?$/, '')` | "双 /api 404"隐患 |
| **插件未使用的后端能力** | `PUT /message-template`、`/:id/sync`、`/batch-sync`、`DELETE /:id` 插件未调用 | "能采集不能同步"体验断裂（同步疑似由 `bd/chrome-extension` 前端页触发，需明确职责） |

### 6.3 优化建议

- **必做**：修复 `batch-videos` 404（后端加路由，或插件改复用 `POST /` 循环）；统一 `apiUrl` 为单一常量（如 `constants.js` 的 `DEFAULT_API_URL`）。
- **配置化**：`popup.js` 的 `FRONTEND_BASE_URL`、跳转路径从 `chrome.storage` 读取；约定 `apiUrl` 始终不含 `/api`，在 `getApiBaseUrl()` 单一出口规范化。
- **代码结构**：抽出共享 `apiClient`（content/background/popup 共用），统一超时/重试/401 处理；统一 `storage` 工具，消除 `storage.js` 与各处重复实现。
- **契约对齐**：`/influencer/:tiktokId` 返回字段（`followers`/`totalLikes`）与插件注释（"FV/GMV/MSS/APV"）语义不一致，建议后端字段名与业务语义对齐。
- **健壮性**：后端 404/错误时插件侧 `catch` 后展示"暂无数据"而非中断；`injected.js` 依赖 TikTok 内部全局变量，TikTok 改版即失效，建议加"抓取失败计数"遥测与手动补录入口；`checkAuthStatus` 仅提示过期未实现自动刷新（Part1 规划的 Refresh Token 未落地）。
- **文档落差**：`TESTING.md` 的"下一步"清单全为未勾选（批量导入、本地缓存、通知等）；Part2 实际实现与计划差异较大（改为写入独立 `tiktok_extension_data` 表）。建议在插件 README 中更新真实状态。
