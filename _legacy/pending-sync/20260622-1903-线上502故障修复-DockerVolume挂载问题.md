## 二、根因分析

### 问题定位流程
1. **本地测试** → 所有API正常（200 OK, 0.16s响应）
2. **确认范围** → 仅线上环境异常
3. **SSH到服务器** → 检查容器状态
4. **发现关键线索** → backend容器处于 `Restarting` 循环状态

### 根因确认
```bash
# 容器日志显示核心错误
docker logs tap-backend --tail 20

# 输出:
Error: Cannot find module 'express'
Require stack:
- /app/server.js
    at Module._resolveFilename (node:internal/modules/cjs/loader:xxxx:xx)
    at Module._load (node:internal/modules/cjs/loader:xxxx:xx)

# 原因链:
docker-compose.yml 配置了 volumes: - ./server:/app/server
    ↓
宿主机 ./server 目录覆盖了容器的 /app/server
    ↓
宿主机没有 node_modules 目录 → 容器的 node_modules 被覆盖
    ↓
启动时找不到 express 模块 → 后端崩溃
```

### 问题配置（已修复前）
```yaml
# docker-compose.yml - backend服务
backend:
  build:
    context: .
    dockerfile: Dockerfile
  volumes:
    - ./server:/app/server   # ← 致命缺陷！
  ports:
    - "3000:3000"
  environment:
    - NODE_ENV=production
    # ...
```

---

## 三、修复方案

### 修复步骤
```bash
# 1. 备份原配置文件
cd /home/ubuntu/tap-system
cp docker-compose.yml docker-compose.yml.bak.$(date +%Y%m%d_%H%M%S)

# 2. 编辑 docker-compose.yml，移除 backend 的 volumes 挂载
# 删除以下内容:
#   volumes:
#     - ./server:/app/server

# 3. 强制删除旧容器
sudo docker rm -f tap-backend

# 4. 使用新配置启动容器
sudo docker compose up -d backend

# 5. 验证容器状态
docker ps | grep tap-backend
# 预期输出: Up xxx seconds (healthy) 或 Up xxx seconds

# 6. 查看日志确认正常启动
docker logs tap-backend --tail 10
# 预期输出:
# MongoDB Connected successfully!
# Server running on port 3000
```

### 修复后配置（正确✅）
```yaml
# docker-compose.yml - backend服务（生产环境正确配置）
backend:
  build:
    context: .
    dockerfile: Dockerfile
  container_name: tap-backend
  restart: unless-stopped
  ports:
    - "3000:3000"
  environment:
    - NODE_ENV=production
    - MONGODB_URI=mongodb://mongodb:27017/tap_system
    - JWT_SECRET=your-jwt-secret-here
    - CORS_ORIGIN=https://tap.lazyfirst.com
  depends_on:
    - mongodb
  networks:
    - tap-network
  # ❌ 注意：绝对不能加 volumes: - ./server:/app/server
```

## 五、经验总结与预防措施

### 🚨 核心教训

#### 1. Docker Volume挂载的陷阱
| 场景 | 是否安全 | 说明 |
|------|----------|------|
| 开发环境挂载代码目录 | ⚠️ 有条件安全 | 必须确保宿主机有完整的 node_modules |
| 生产环境挂载代码目录 | ❌ 绝对禁止 | 会导致依赖丢失、版本不一致 |
| 挂载配置文件（.env） | ✅ 安全 | 用于外部化配置 |
| 挂载静态资源（SSL证书、上传目录） | ✅ 安全 | 必需的数据持久化 |
| 挂载数据卷（数据库数据） | ✅ 安全 | 数据持久化必需 |

#### 2. 正确的生产部署架构
```
Dockerfile（构建镜像）:
  COPY server/ /app/server/
  WORKDIR /app
  RUN npm install --omit=dev   ← 依赖安装进镜像

docker-compose.yml（运行容器）:
  backend:
    build: .
    ports: ["3000:3000"]
    # 不挂载代码目录！
    # 只挂载必要的配置和数据:
    # volumes:
    #   - ./ssl:/etc/nginx/ssl  ← SSL证书（如需）
    #   - uploads:/app/uploads  ← 用户上传文件（如需）
```

#### 3. 排查思路速查表
| 现象 | 排查方向 | 关键命令 |
|------|----------|----------|
| 502 Bad Gateway | 后端服务未运行/崩溃 | `docker ps -a`, `docker logs <container>` |
| 容器 Restarting 循环 | 启动失败/依赖缺失 | `docker logs <container> --tail 50` |
| Cannot find module | node_modules 缺失或损坏 | `docker exec <container> ls -la /app/node_modules` |
| 本地正常线上异常 | 配置差异/环境变量 | 对比本地和线上 docker-compose.yml |
| Nginx 502 | 后端端口不通/容器未启动 | `curl localhost:3000`, `docker ps` |
