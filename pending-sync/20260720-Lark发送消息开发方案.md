# Lark 发送消息开发方案（卖家侧）

> 制定日期：2026-07-20
> 目标：在 LazyFirst 现有 LINE 卖家侧能力基础上，用 **Lark 国际版** 平行复刻「绑定引导」+「申样通知」两项能力。

---

## 一、已确认决策点（来自主人）

| 项 | 决策 |
|----|------|
| 功能范围 | 仅卖家侧：①绑定引导 ②申样通知（**不做**达人侧活动/新品推送） |
| 绑定方式 | **BD 手动录入 Lark ID**（无 Webhook、无绑定码流程） |
| 平台 | **Lark 国际版**，域名 `https://open.larksuite.com`，需 `APP_ID` / `APP_SECRET` |

---

## 二、整体架构对比（LINE vs Lark）

| 维度 | LINE（现状） | Lark（本方案） |
|------|--------------|----------------|
| 配置 | `server/config/line.js` | `server/config/lark.js`（新建） |
| 客户端 | `server/line/client.js` | `server/lark/client.js`（新建） |
| 绑定服务 | `server/line/bindingService.js`（生成绑定码+Webhook解析） | `server/lark/bindingService.js`（BD 录入+发测试消息校验） |
| 推送服务 | `server/line/pushService.js` | `server/lark/pushService.js`（新建） |
| 路由 | `server/routes/line.js` | `server/routes/lark.js`（新建） |
| 模型字段 | `ShopContact.lineUserId` / `lineBindStatus` | `ShopContact.larkOpenId` / `larkBindStatus` 等（新增） |
| 前端弹窗 | `LineBindingDialog.vue`（绑定码） | `LarkBindingDialog.vue`（录入 open_id，新建） |
| Webhook | 有（`webhookHandler.js`） | **无**（本期不做事件订阅） |
| 申样触发 | `server/routes/samples.js` 异步 push | 同文件增加 Lark 分支 |

**核心差异**：Lark 不需要 Webhook 和绑定码，绑定由 BD 在后台直接录入卖家 Lark `open_id`，后端用「发一条测试消息」来校验 ID 有效性，成功即写入绑定。

## 三、详细改造清单

### 1. 后端配置（新建）
**`server/config/lark.js`**
```js
const isConfigured = !!(process.env.LARK_APP_ID && process.env.LARK_APP_SECRET);
module.exports = {
  isConfigured,
  appId: process.env.LARK_APP_ID,
  appSecret: process.env.LARK_APP_SECRET,
  baseUrl: 'https://open.larksuite.com',
  appType: 'international', // 国际版；若以后接飞书国内版改 open.feishu.cn
  receiveIdType: 'open_id',
  tokenCacheTtl: 60 * 60 * 1000 - 5 * 60 * 1000, // 提前5分钟过期
};
```

### 2. Lark 客户端（新建）
**`server/lark/client.js`**
- `getTenantAccessToken()`：调用 `POST /open-apis/auth/v3/tenant_access_token/internal`，内存缓存 token（按 `tokenCacheTtl`）。
- `sendMessage(openId, msgType, contentObj)`：调用 `POST /open-apis/im/v1/messages?receive_id_type=open_id`，header `Authorization: Bearer <token>`，body `{ receive_id, msg_type, content }`（`content` 需 `JSON.stringify`）。
- `sendInteractiveCard(openId, cardJson)`：封装 interactive 卡片发送。
- `verifyUser(openId)`：用「发一条测试消息」校验 open_id 是否有效（见第五节）。

> 注意：Lark 国际版发消息的 `content` 必须是 **JSON 字符串**（`JSON.stringify`），不是对象。

### 3. 绑定服务（新建）
**`server/lark/bindingService.js`**
- `bindSellerContact(contactId, openId, operatorId)`：
  1. 调 `client.verifyUser(openId)` 发测试消息；
  2. 成功 → `ShopContact.updateOne({ _id: contactId }, { larkOpenId: openId, larkBindStatus: 'bound', larkBoundAt: new Date(), larkBoundBy: operatorId })`；
  3. 失败 → 抛错返回「open_id 无效或机器人不可见」。
- `unbindSellerContact(contactId)`：清空 `larkOpenId / larkBindStatus` 等字段。
- `getBindStatus(contactId)`：返回当前绑定状态。

### 4. 推送服务（新建）
**`server/lark/pushService.js`**
- `pushSampleApprovalCard({ openId, productName, influencerName, sampleId, approvalUrl })`：构造 interactive 卡片并发送（卡片 JSON 见第六节）。

### 5. 路由（新建）
**`server/routes/lark.js`**（挂载在 `/api/lark`）
| Method | Path | 说明 |
|--------|------|------|
| POST | `/api/lark/bind` | BD 录入 open_id 绑定卖家联系人 |
| POST | `/api/lark/unbind` | 解绑 |
| GET | `/api/lark/status/:contactId` | 查绑定状态 |
| POST | `/api/lark/test` | 给指定 open_id 发测试消息（手动验证） |
| POST | `/api/lark/push-sample` | 手动补推申样通知（可选，便于排查） |

鉴权沿用 `authenticate` + `authorize('shops:update' 或 'samplesBd:update')`，与 LINE 路由一致。

### 6. 模型（修改）
**`server/models/ShopContact.js`** 新增字段（与现有 `lineUserId` 并列，均为可选，旧数据无迁移）：
```js
larkOpenId:   { type: String, default: '' },
larkBindStatus: { type: String, default: '' }, // 'bound' | ''
larkBoundAt:  { type: Date },
larkBoundBy:  { type: mongoose.Schema.Types.ObjectId, ref: 'User' },
```

### 7. 申样触发（修改）
**`server/routes/samples.js`（第 558–580 行 LINE 通知块之后追加 Lark 分支）**
```js
// Lark 通知：仅通知卖家（与 LINE 并列，独立 try/catch 互不阻塞）
if (larkConfig.isConfigured && product?.shopId) {
  setTimeout(async () => {
    try {
      const seller = await ShopContact.findOne({
        shopId: product.shopId,
        larkOpenId: { $exists: true, $ne: '' }
      });
      if (seller?.larkOpenId) {
        await larkPush.pushSampleApprovalCard({
          openId: seller.larkOpenId,
          productName, influencerName, sampleId,
          approvalUrl: `${process.env.FRONTEND_BASE_URL}/samples/${sampleId}`
        });
        console.log(`[Lark] 申样审批通知推送成功 → 卖家 (${seller.larkOpenId})`);
      }
    } catch (e) { console.warn('[Lark] 通知卖家申样失败:', e.message); }
  }, 200);
}
```
> LINE 与 Lark 双渠道可同时触发：卖家若两边都绑定，则两边都收到；任一边失败不影响另一边。

### 8. 路由挂载（修改）
**`server/app.js` 或 `server/index.js`**：`app.use('/api/lark', larkRoutes)`（与现有 `/api/line` 并列）。

### 9. 前端（新建 + 修改）
**新建 `frontend/src/components/LarkBindingDialog.vue`**
- 弹窗内一个输入框：卖家 Lark `open_id`（placeholder 提示卖家如何在 Lark 客户端复制 open_id）。
- 确定 → `POST /api/lark/bind`；成功提示「已绑定 Lark」并 emit `bound-changed`。
- 显示当前绑定状态 + 解绑按钮。
- 复制 `LineBindingDialog.vue` 骨架，去掉绑定码相关逻辑。

**修改 `frontend/src/views/products/ShopTab.vue`（卖家联系人页）**
- 第 250–256 行 `LineBindingDialog` 下方新增并排的 `LarkBindingDialog`（同样 `role="shopContact"`, `:target-id="lineBindingTarget.id"`）。
- 第 283 行 import 增加 `LarkBindingDialog`。
- 新增 `showLarkBinding` ref、`openLarkBinding(row)`、`onLarkBoundChanged()`（参照现有 `openLineBinding` / `onLineBoundChanged`）。
- 联系人表格增加「Lark 状态」列（绑定/未绑定）。

### 10. 环境变量（修改）

## 四、Lark 国际版关键 API 端点

| 用途 | Method & URL | 说明 |
|------|--------------|------|
| 获取 tenant_access_token | `POST https://open.larksuite.com/open-apis/auth/v3/tenant_access_token/internal` | body `{ app_id, app_secret }` → `{ tenant_access_token, expire }` |
| 发送消息 | `POST https://open.larksuite.com/open-apis/im/v1/messages?receive_id_type=open_id` | header `Authorization: Bearer <token>`；body `{ receive_id, msg_type, content }` |
| （可选）查用户 | `GET https://open.larksuite.com/open-apis/contact/v3/users/:user_id?user_id_type=open_id` | 需通讯录权限；本期用「发测试消息」替代，避免额外权限 |

**所需应用权限（国际版后台开启）**：
- `im:message`（发送消息）
- `im:message:send_as_bot`（以机器人身份发送）
- 应用需添加为「机器人」，且卖家账号与机器人可见（企业内分发或应用市场可见）。

## 五、绑定流程设计（BD 手动录入）

```
BD 在 ShopContact 编辑弹窗点「绑定 Lark」
  → 输入卖家 open_id → 确定
  → POST /api/lark/bind { contactId, openId }
  → 后端 larkClient.verifyUser(openId):
        发送一条 "绑定成功，您已接入 LazyFirst 通知" 的测试消息
        ├─ 成功 → 写入 larkOpenId / larkBindStatus='bound' / larkBoundAt / larkBoundBy
        └─ 失败(用户不存在/机器人不可见) → 返回错误，前端提示「open_id 无效」
  → 前端刷新状态列
```

> 校验采用「发测试消息」而非通讯录 API，原因是国际版应用默认即有 `im:message` 权限即可发消息，无需额外申请通讯录读权限，落地更快、权限更省。

## 六、申样通知卡片（interactive）

`msg_type: "interactive"`，`content` 为 `JSON.stringify` 的卡片对象（Lark 旧版 card schema，国际版兼容稳定）：
```json
{
  "type": "card",
  "data": {
    "card": {
      "config": { "wide_screen_mode": true },
      "header": {
        "title": { "tag": "plain_text", "content": "新的样品申请待您审批" },
        "template": "orange"
      },
      "elements": [
        {
          "tag": "div",
          "text": {
            "tag": "lark_md",
            "content": "**达人**：{{influencerName}}\n**样品**：{{productName}}\n**申请编号**：{{sampleId}}"
          }
        },
        {
          "tag": "action",
          "actions": [
            {
              "tag": "button",
              "text": { "tag": "plain_text", "content": "前往审批" },
              "type": "primary",
              "url": "{{approvalUrl}}"
            }
          ]
        }
      ]
    }
  }
}
```
> 备注：若后续启用 Lark 新版卡片（ui 字段），再平滑升级卡片结构，推送服务封装层不变。

## 七、错误处理与降级
- `larkConfig.isConfigured` 为 false（未配置凭证）→ 跳过 Lark 分支，不影响 LINE 与现有流程。
- 绑定/推送失败仅 `console.warn`，不阻断主流程（与 LINE 现有做法一致）。
- 申样通知 LINE 与 Lark 双通道独立 `try/catch`，互不影响。

## 八、安全与权限
- `/api/lark/*` 全部走 `authenticate` + 角色鉴权（复用现有 `shops`/`samplesBd` 权限）。
- `larkOpenId` 仅后台 BD 可写入，卖家无前端写入入口。
- 凭证仅存服务端环境变量，不进前端、不进 git。

## 九、上线步骤（概览，涉及线上需主人同意）
1. 代码合入后，`docker build` 加 `--no-cache` 重新构建后端镜像（遵 memory 铁律）。
2. 生产 `.env` 增加 `LARK_APP_ID` / `LARK_APP_SECRET`（主人提供凭证）。
3. Lark 国际版后台应用开启 `im:message` / `im:message:send_as_bot`，添加为机器人。
4. `docker compose restart backend` + 前端重新构建部署。
5. 验证：BD 录入一个测试 open_id → 收到测试消息 → 发起一条测试申样 → 卖家收到审批卡片。

## 十、风险与待确认
- **open_id 录入友好度**：BD 需卖家提供 open_id（非邮箱）。若主人希望用邮箱录入，可改 `receive_id_type=email`（需确认应用权限）。
- **机器人可见性**：国际版应用需卖家企业内可见，否则发消息失败；上线前需主人在 Lark 后台确认分发范围。
- **卡片 schema**：先用旧版 card JSON，稳定兼容；新版 `ui` 卡片后续按需升级。
- **凭证**：`APP_ID/APP_SECRET` 由主人向 Lark 国际版申请，小垃圾不擅自猜测或记忆任何密钥。
