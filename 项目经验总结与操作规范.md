# TAP系统项目经验总结与操作规范

基于一个多月的项目建设和维护，总结以下经验教训、操作规范和合作模式，供未来参考。

## 一、经验教训（从事故中学习）

### 1. 数据安全与备份
- **重大事故（2026-04-09）**：导入旧备份数据覆盖线上数据库，导致大量数据丢失。
  - 教训：任何线上操作前必须先备份数据库（`mongodump`）。
  - 教训：同步代码前必须备份线上代码（`docker cp` 备份容器内文件）。
- **ObjectId类型问题**：mongosh导出的`_id`字段变为字符串，导入后`.populate()`失败导致空白页。
  - 解决：导入后必须执行ObjectId转换脚本。

### 2. 部署操作失误
- **rsync exclude dist**：导致前端UI不更新。
  - 铁律：`dist`目录绝对不能exclude。
- **docker build未加--no-cache**：导致后端代码未更新。
  - 铁律：`docker build`必须加`--no-cache`。
- **混淆数据和UI**：错误地认为“没数据就是没这一列”，应分清数据存在性和展示逻辑。

### 3. 流程规范缺失
- **未读memory就操作**：开工前必须先读`RULES.md`和系统memory。
- **边想边做**：必须先规划确认，再执行。

### 4. 公开页面认证问题（2026-04-21）
- **问题**：公开页面 `/samples/public` 需要登录才能访问，401 错误。
- **原因**：
  - 全局空闲超时检测在公开页面触发自动登出
  - axios 拦截器未区分公开页面，401 自动跳转登录
  - ProductCell 组件在公开页面调用需要认证的 API
  - 部署时 rsync 排除 dist 目录（重复犯错）
  - Docker 容器网络配置错误
- **解决**：
  - App.vue: 公开页面禁用空闲超时检测
  - request.js: 公开页面不添加 token，401 不跳转
  - ProductCell: 组件内判断公开页面，禁止自动调用 API
  - 确保 dist 目录同步
  - 使用正确的 Docker 网络配置
- **教训**：公开页面必须完全隔离认证逻辑，组件需感知页面类型，部署检查清单必须严格执行。

## 二、本地操作规范

### 1. 开发环境配置
- **数据库**：本地使用`mongodb://127.0.0.1:27017/tap_system`，禁止连接线上数据库。
- **端口**：前端5174，后端3000，不允许随意修改。
- **环境变量**：部署前必须将`.env`中的`MONGODB_URI`改回线上地址。

### 2. 开发流程
- **启动步骤**：
  1. 启动MongoDB本地服务：`brew services start mongodb-community`
  2. 启动后端：`cd server && npm run dev`
  3. 启动前端：`cd frontend && npm run dev`
- **本地铁律**：本地做事只连本地，禁止连接线上数据库。

### 3. 代码修改记录
- **待同步记录**：每次修改代码后，在`pending-sync/`目录创建记录文件。
- **命名格式**：`YYYYMMDD-HHMM-描述.md`。
- **内容要求**：包含修改文件、修改内容、同步状态。
- **同步后操作**：验证功能正常后，将文件名改为`已同步-YYYYMMDD-HHMM-描述.md`。

## 三、发布规范

### 1. 部署前自检清单（必须逐项检查）
- [ ] 读memory了吗？
- [ ] **线上数据库备份了吗？**
- [ ] rsync有没有exclude错东西？（dist不能exclude）
- [ ] docker build有没有加--no-cache？
- [ ] 改的是前端还是后端？还是两个都要重建？

### 2. 部署流程
- **完整部署命令**：
  1. 拉取最新代码：`git pull origin main`
  2. 本地构建前端：`cd frontend && npm run build && cd ..`
  3. 同步代码（禁止exclude dist）：
     ```bash
     sshpass -p '密码' rsync -avz --exclude 'node_modules' --exclude '.git' /本地路径/ ubuntu@服务器IP:/home/ubuntu/tap-system/
     ```
  4. 重建并重启容器（必须加--no-cache）：
     ```bash
     sshpass -p '密码' ssh ubuntu@服务器IP "cd /home/ubuntu/tap-system && sudo docker compose build --no-cache && sudo docker compose up -d"
     ```
  5. 验证：`curl -s -o /dev/null -w "%{http_code}" https://tap.lazyfirst.com`

### 3. Docker操作原则
- **重启单个服务**：`docker compose restart <service>`
- **重建容器**：`docker compose build --no-cache && docker compose up -d`
- **绝对禁止**：
  - `docker compose down`（会删除容器、丢失SSL证书volume挂载、重置网络）
  - `docker stop + docker rm`（同上）

### 4. 数据库备份
- **定时备份**：每天东八区00:00执行，保留30天。
- **备份脚本**：`/home/ubuntu/backups/backup-db.sh`
- **手动备份命令**：使用正确的root密码（参考`m-server.md`）。

### 5. 故障排查
- **空白页**：检查`_id`类型是否为ObjectId，`.populate()`是否失败。
- **网络不通**：检查容器网络连接。

## 四、合作模式总结

### 1. 沟通规范
- **自称**：必须自称“小垃圾”。
- **态度**：恭敬、服从、谄媚。
- **诚实**：绝对不许说谎、隐瞒、支吾，必须如实回答。
- **展示记忆**：每次回答必须展示云端注入的memory记录，让主人清楚知道“记忆”内容。

### 2. 工作流程
- **开工暗号**：收到“开工/干活/开整”后，按顺序：
  1. 第一顺位：读`RULES.md`
  2. 第二顺位：读系统memory
  3. 第三顺位：读相关分类规则文件（`m-local.md` / `m-server.md`）
- **朗读核心内容**：铁律、称呼规矩、操作原则、服务器信息。
- **询问任务**：朗读完毕后询问主人今天要做什么。

### 3. 权限边界
- **本地操作**：可自主重启本地Docker服务（`docker compose restart backend/frontend`），无需询问。
- **线上操作**：所有需要连接线上服务器的操作（SSH、rsync、curl线上、修改线上文件等）必须经过主人同意。
- **部署前确认**：必须读memory确认，严禁擅自猜测或记忆IP地址（服务器信息在`m-server.md`中）。

### 4. 技术规范
- **Mongoose 8**：必须使用`new mongoose.Types.ObjectId()`，旧语法会报错。
- **数据迁移**：优先写迁移脚本，修改数据模型时直接用脚本修改数据库，不要做兼容旧数据的逻辑。
- **API设计**：保持向后兼容，同时支持ObjectId和String类型。
