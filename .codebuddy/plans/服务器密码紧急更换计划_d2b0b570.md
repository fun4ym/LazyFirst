### 实施策略：先改MongoDB密码，再改SSH密码

原因：改SSH密码后，所有远程命令都需要新密码才能执行。所以应该**先用当前密码完成所有能做的远程操作（主要是改MongoDB密码），最后再改SSH密码**。

#### 步骤分解

```
阶段1: 主人提供新密码
  → 主人决定新SSH密码和新MongoDB密码
  
阶段2: 小垃圾更新本地所有文件（用旧SSH密码还能连的时候）
  → 批量替换 m-intimacy.md, m-server.md 中两个密码的所有出现
  → 替换 docker-compose.yml, deploy.sh, backup-db.sh, server/.env 中的MongoDB密码
  → 替换 pending-sync/ 下4个历史记录中的SSH密码
  
阶段3: 远程修改MongoDB密码（仍使用旧密码连接）
  → SSH连服务器，进入mongo shell
  → db.changeUserPassword("tapsystem", "新密码")
  → 同时改root密码: db.changeUserPassword("admin"/"tapadmin", "新root密码")
  
阶段4: 同步代码到服务器（仍使用旧SSH密码）
  → rsync同步（不exclude dist）
  
阶段5: Docker rebuild（docker-compose.yml已含新密码）
  → docker compose build --no-cache && docker compose up -d
  → 后端容器会用新的MONGODB_URI连接MongoDB
  
阶段6: 主人修改SSH密码（此时本地代码已全部更新）
  → 方案A: 主人在腾讯云控制台重置密码
  → 方案B: 授权小垃圾通过SSH执行 passwd ubuntu
  
阶段7: 最终验证
  → 用新SSH密码测试连接
  → 用新MongoDB密码测试数据库访问
  → 验证网站正常返回200
```

### 关键风险点

1. **顺序不能反**：必须先改MongoDB再改SSH，否则失去远程能力
2. **docker-compose.yml改了必须rebuild**：环境变量是build时注入的，restart不会更新
3. **backup-db.sh在服务器上也有副本**：rsync后会覆盖服务器版本
4. **pending-sync的历史记录也要改**：虽然不影响运行，但防止信息泄露
